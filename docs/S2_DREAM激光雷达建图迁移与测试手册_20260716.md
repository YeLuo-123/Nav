# S2 DREAM 激光雷达建图迁移与测试手册

## 1. 当前结论

S2 底盘的激光雷达数据已经能够在 DREAM 主机端稳定接收，现有 G1 二维雷达建图的核心逻辑可以复用，但输入改为 S2 的双雷达融合点云和底盘里程计。

结合《S2产品说明书V2.0 Beta》和实机接口，当前底盘使用 Livox Mid-360S 系列雷达；实机对外发布 `lidar_front`、`lidar_back` 两路约 2 万点/帧的点云，并提供约 4 万点/帧的 `base_link` 融合点云。因此 DREAM 直接消费融合点云，不重复拼接前后雷达外参。

当前推荐链路：

```text
前/后 Livox 点云
  -> S2 底盘融合为 base_link 点云
  -> DREAM 主机按高度和距离切片
  -> 基于 /controller/odom 投影到 odom 坐标系
  -> 射线更新二维占据栅格
  -> 噪点过滤
  -> PGM/YAML/JSON、ROS OccupancyGrid、HTTP、Rerun
```

本迁移不控制底盘运动，只读取传感器和里程计数据。

## 2. 网络和 ROS 2 参数

《通用机器人使用说明文档26.0》确认：

- S100 的 DDS 数据网口为 `eth1=192.168.127.10/24`。
- 主机连接底盘的网卡为 `eno1`，本项目设置为 `192.168.127.100/24`。
- `ROS_DOMAIN_ID=0`。
- `RMW_IMPLEMENTATION=rmw_cyclonedds_cpp`。
- CycloneDDS 必须绑定主机的 `eno1`，配置文件为：
  `DREAM/configs/robots/s2_cyclonedds.xml`。

`6.6.6.6` 是另一管理侧地址，不是本次 ROS 2 DDS 接入地址。

## 10. 当前算法边界

当前实现是“底盘里程计约束下的点云二维栅格融合”，不是完整的回环 SLAM：

- 短距离实机测试和为导航提供干净局部二维图可用。
- 长距离运行仍可能随 `/controller/odom` 漂移。
- 厂商内置 `/cloudRegistered` 和 `/global_map` 启动后，可切换到其定位建图结果，或后续接入成熟的 LIO/SLAM 做回环修正。
- 文档中的 `/global_map` 类型是三维 `PointCloud2`，不是 `OccupancyGrid`；当前不要使用 `start-native-map`，除非厂商后续提供真实二维占据图话题。
- 高度切片 `0.25-1.20 m` 是当前实测初值，应根据底盘实际障碍物漏检或桌面噪点情况调整，不能把所有三维表面无差别压平。

- `/s2_lidar_slam/point_cloud`：由建图进程在主机本地重发的前后雷达融合三维点云。
- `/s2_lidar_slam/odom`：转换为标准四元数后的底盘位姿。
- `/robot_description`：由 `S2_3DURDF` 和 STL 加载的真实三维底盘模型。
- `/joint_states`：四个轮子的显示关节状态，当前固定为零位，不参与控制。
- `/s2_lidar_slam/configured_footprint`：Nav2 实际使用的带 padding 碰撞轮廓。
- `odom -> base_link`：仅用于本次可视化的 TF。

使用本地点云话题是为了让跨网线的大体量点云只传输一次，同时统一底盘和主机时间戳。RViz 使用倾斜的 Orbit 三维视角；鼠标左键旋转、中键平移、滚轮缩放。

实测 STL 外包围盒约为 `0.670 x 0.548 x 0.390 m`。Nav2 基础 footprint
为 `0.86 x 0.66 m`，再加四周 `0.03 m` padding，因此 RViz 黄色线框最终显示
`0.92 x 0.72 m` 的碰撞边界。黄色线框比实体模型大是预期的安全余量，不是模型
缩放错误。

关闭 RViz 后，脚本会自动停止其 TF/里程计桥接进程和模型发布器，不影响二维建图进程。

旧会话一旦已经出现扇形重影，不能通过修改参数从累计 log-odds 中可靠删除错误
障碍。必须停止旧进程，并按“终端 2”创建新的 `SESSION` 后重新建图。

S2 的 `/controller/odom` 不是标准四元数：航向角直接保存在
`pose.pose.orientation.w`。本流程已经固定使用
`S2_ODOM_YAW_MODE=orientation_w`，不要改回 `auto`，否则航向接近
`+/-1 rad` 时可能被误判成四元数并产生旋转重影。

点云比最新 odom 约晚到主机 `0.12 s`。建图器现在按消息时间戳从 odom
历史队列选择最近位姿，实测匹配误差约 `0.002 s`，然后将点云转换到
`odom` 坐标系后再发布给 RViz 和 Nav2。

默认 `S2_NAV2_ENABLE_MOTION=true`。启动器会通过底盘官方软手柄接口
`/api/AMR/SetManualMode` 切换到软手动模式，通过 `/api/AMR/StopParking`
退出驻车，并把 Nav2 速度发送至
`ws://192.168.127.10:6001/api/AMR/ManualMove`。在 RViz 中使用 `Nav2 Goal`
选点后，底盘会真实移动。

首次实机测试默认限制为前后 `0.12 m/s`、横移 `0.10 m/s`、旋转
`0.30 rad/s`。测试人员必须站在急停旁，清空底盘周围至少 1 m 区域，并先选
`0.3-0.5 m` 内的自由目标点。

该启动文件连接：

- 全局地图：`/s2_lidar_slam/map`，5 cm 栅格，`odom` 坐标系。
- 局部障碍：`/s2_lidar_slam/point_cloud`，三维 PointCloud2 高度切片。
- 里程计：`/s2_lidar_slam/odom`。
- TF：`odom -> base_link`。
- 全局规划器：Navfn A*。
- 局部控制器：DWB，全向底盘参数。
- 平滑后速度：`/s2_nav2/cmd_vel`。
- 速度监视：`/s2_nav2/cmd_vel_preview`。
- 底盘控制：厂商软手柄 WebSocket；不与 `/move/AutoMoveCmd` 的 100 Hz
  厂商导航零速发布者竞争。

### 12.5 RViz 小范围实机移动

1. 确认底盘无急停、无驻车，周围无人和杂物。
2. 启动建图与 Nav2，等待五个 lifecycle 节点均为 `active [3]`。
3. 在 RViz 顶部选择 `Nav2 Goal`。
4. 只在机器人附近明确的白色自由区按下并拖动，箭头方向是目标朝向。
5. 第一次目标距离控制在 `0.3-0.5 m`，观察全局路径、局部路径和底盘运动。
6. 需要立即停止时，按底盘急停；正常停止可在 Navigation 2 面板点击 Cancel，
   或在 Nav2 启动终端按 `Ctrl+C`。

## 14. 无线通信方案

### 14.1 当前限制

底盘文档只定义 `eth0/eth1`，ROS 2 DDS 固定绑定底盘 `eth1=192.168.127.10/24`，没有提供可直接启用的底盘 Wi-Fi ROS 接口。因此不能只拔掉网线并让主机连接普通 Wi-Fi。

### 14.2 推荐硬件拓扑

在底盘上增加支持透明二层桥接的 5 GHz/Wi-Fi 6 网桥或车载路由器：

```text
S2 eth1 192.168.127.10
  -> 车载透明无线网桥
  -> 5 GHz/Wi-Fi 6
  -> 主机无线网卡 192.168.127.100
```

要求：

- 使用 bridge/WDS/透明客户端模式，不能使用普通 NAT 客户端模式。
- AP 必须关闭无线客户端隔离。
- 主机和 S2 必须处于同一个二层广播域。
- 建议固定 5 GHz 信道，避免 2.4 GHz。
- 设备实际 TCP/UDP 吞吐应稳定高于 150 Mbps，推荐 Wi-Fi 6。
- 演示区域应提前进行覆盖测试和持续丢包测试。

更简单的现场方案是在底盘上安装一个小型 Wi-Fi 6 路由器作为独立 AP：S2 网线接其 LAN 口，主机直接连接机器人专用 SSID。这样不依赖场馆现有无线网络。

### 14.3 主机无线配置

无线网桥完成后，在主机 Wi-Fi 网卡增加 S2 子网地址。当前主机无线接口为 `wlxfc221c100c12`：

```bash
sudo ip address add 192.168.127.100/24 dev wlxfc221c100c12

ping 192.168.127.10
```

如果有线 `eno1` 仍配置着同一地址，切换无线前先断开有线连接，避免同一子网同时出现在两个接口：

```bash
nmcli connection down DREAM-S2-test
```

无线 DDS 配置：

```bash
export ROS_DOMAIN_ID=0
export RMW_IMPLEMENTATION=rmw_cyclonedds_cpp
export ROS_DISABLE_DAEMON=1
export CYCLONEDDS_URI=file:///home/fq/BJHYZJ_FQ/DREAM/configs/robots/s2_cyclonedds_wifi.xml

ros2 topic hz /driver/lidar/point_cloud/Data
ros2 topic hz /controller/odom
```

### 14.4 带宽建议

融合点云约 `40128` 点/帧、`10 Hz`，按照当前字段布局，纯点云载荷约为 `80-90 Mbps`，加上 DDS 分片、协议开销、重传竞争和其他话题后，应按 `100-130 Mbps` 规划。

因此：

- 信号优秀、专用 5 GHz/Wi-Fi 6 网络下，可以直接无线传输融合点云。
- 不建议同时传输前雷达、后雷达和融合点云，只订阅融合话题。
- 公共或拥挤 Wi-Fi 下不适合作为正式 Demo 的唯一链路。
- 最稳定的后续架构是在底盘端完成体素降采样或二维建图，只无线发送降采样点云、OccupancyGrid 和里程计，可将带宽降低一个数量级。
